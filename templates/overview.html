BlindBuild

Round 2: Problem Brief for API Architecture

Participants must design the backend API structure required to support frontend workflows. Each question below defines application behavior, user actions, and design expectations.

1. Event Registration & Ticket System

Problem Statement

BlindBuild organizers need a platform where participants can register for events and receive digital tickets.

Scenario Context

The flow should cover a participant journey from onboarding to participation completion, including discovering event options, completing enrollment, confirming status, receiving a ticket identity, and handling cancellation when needed.

Task Focus

Design the complete API workflow, including resources, methods, payloads, and response formats.

Concepts Tested

API architecture, workflow design, and system thinking.

2. Team Based Hackathon Portal

Problem Statement

Students should be able to form teams and complete hackathon registration as a unit.

Scenario Context

The solution should represent team lifecycle management, where teams are formed, membership changes are tracked, details can be revised over time, and registration outcomes remain visible for verification.

Task Focus

Define a clear resource hierarchy for team and member operations with structured request and response models.

Concepts Tested

Resource design, API hierarchy, and CRUD operations.

3. Workshop Slot Booking System

Problem Statement

Users need to reserve workshops where each session has limited slot availability.

Scenario Context

Participants should be able to understand workshop options, review real-time seat capacity, reserve a position, reverse decisions when required, and verify whether the booking state is active.

Task Focus

Design APIs that manage nested workshop-slot relationships and booking lifecycle actions.

Concepts Tested

REST resource structure, nested resources, and end-to-end flow design.

4. Event Check-In System

Problem Statement

Participants must check in on event day using ticket IDs to validate entry and attendance.

Scenario Context

Check-in should establish a reliable sequence from ticket validation to participant retrieval, attendance confirmation, badge assignment, and final status visibility for event operations.

Task Focus

Build a robust check-in workflow with clear verification and attendance state transitions.

Concepts Tested

Validation logic, event workflow sequencing, and consistency handling.

5. Event Feedback & Rating System

Problem Statement

Participants should provide feedback and ratings after event completion.

Scenario Context

The platform should support post-event experience capture where responses can be revisited, improved, removed when necessary, and combined into meaningful aggregate sentiment indicators.

Task Focus

Design CRUD-ready feedback APIs and rating aggregation behavior with predictable response formats.

Concepts Tested

CRUD API design, data aggregation, and post-event analytics flow.

6. Lost & Found Campus Portal

Problem Statement

During large events, users should report lost items while others can report found items and support item recovery.

Scenario Context

The workflow should handle reporting, discovery, matching, claiming, moderation approval, and closure so that item return is traceable from first report to final handover.

Task Focus

Design a multi-actor API workflow with clear claim states, role responsibilities, and status transitions.

Concepts Tested

Workflow modeling, resource linking, and status management.

Instruction: Derive the complete backend API contract for each question without relying on pre-listed endpoint patterns. Your design should include resources, HTTP methods, required payload fields, and response structures.
